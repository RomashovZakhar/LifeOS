# 04b — Data model

Модель данных пользователь заранее не задаёт. Твоя задача — предложить варианты.

## Вход

- `docs/DATA.md`, `TECH.md`, `PRODUCT.md`
- `analysis/04_information_architecture.md`
- предыдущие analysis-артефакты и скриншоты (типы трекеров эталона)

## Задача

Предложи **2–3 варианта** локальной схемы данных для V1:

- Habits
- Workouts
- Morning

Учти offline-first, годы хранения, отсутствие бэкенда.

## Для каждого варианта опиши

- сущности и связи
- как моделируются типы значений (completion / time / count / …)
- как модули делят общие примитивы (если делят)
- хранилище (IndexedDB vs alternatives) и почему
- плюсы / минусы / риски
- миграции и эволюция схемы

## Правила

- Не выбирать вариант молча.
- Не писать код.
- Не упрощать до «трёх независимых приложений в одном store», если это ломает единый язык продукта — объясни trade-off.
- Задавай вопросы, если без ответа пользователя варианты бессмысленны.

## Выход

`analysis/04b_data_model.md`

Структура файла:

1. Варианты A/B/(C)
2. Рекомендация агента (явно как рекомендация, не решение)
3. Секция «Утверждено» — пустая, заполняется после выбора пользователя

Стоп. Жди выбора пользователя. После выбора — дополни тот же файл секцией «Утверждено» с финальной схемой.
